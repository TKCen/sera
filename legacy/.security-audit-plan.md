# Security Audit Plan — SERA API Routes Layer

## Scope

Audit the SERA core API routes layer (core/src/routes/_.ts and core/src/auth/_.ts) for:

1. Authentication gaps (endpoints missing auth checks, bypass paths)
2. Input validation gaps (missing zod validation, SQL injection vectors)
3. Error handling (stack trace leaks, internal details exposed)
4. Authorization gaps (missing permission checks per endpoint)
5. Rate limiting gaps (endpoints without rate limits)

## Key Files to Examine

- core/src/index.ts (route mounting, middleware ordering)
- core/src/auth/authMiddleware.ts (auth enforcement)
- core/src/routes/\*.ts (all 40+ route files)

## Investigation Steps

### Phase 1: Auth Middleware & Access Control (READ-ONLY)

1. Read authMiddleware.ts to understand auth enforcement mechanism
2. Read index.ts to verify middleware ordering on critical routes
3. Scan all route files for:
   - Routes that should require auth but don't
   - Routes that have conditional auth
   - Routes that check specific roles/permissions
4. Check for OIDC vs API key auth gaps
5. Verify session token validation

### Phase 2: Input Validation Gaps (READ-ONLY)

1. Grep for zod schema usage across routes
2. Scan for missing body/param validation, SQL injection risks, file upload validation

### Phase 3: Error Handling (READ-ONLY)

1. Check for stack trace exposure and sensitive data in error messages

### Phase 4: Authorization Checks (READ-ONLY)

1. Verify permission checks on protected endpoints

### Phase 5: Rate Limiting (READ-ONLY)

1. Check middleware and identify endpoints without rate limits

### Phase 6: Dependency Audit (READ-ONLY)

1. Run npm audit on core/package.json

### Phase 7: Secrets Scan (READ-ONLY)

1. Grep for hardcoded secrets

## Expected Deliverable

Structured security review report with findings grouped by severity and OWASP category, each with location, exploitability, blast radius, and remediation steps.
